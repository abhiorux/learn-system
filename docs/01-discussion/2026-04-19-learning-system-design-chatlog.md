## 讨论结论

### 总体结论：先实现 Phase 1，以数据模型设计为起点

推荐引擎和图谱可视化的价值完全依赖数据模型的质量——如果知识节点的 schema 定错了，后面改代价极高。数据模型是整个系统的根基。

### 结论分两步走

**Step 1 — 定义知识节点 schema（YAML 格式）**

先回答关键问题：
- 节点必须有哪些字段？（id、title、difficulty、prerequisites、resources...）
- 依赖关系怎么表示？（显式 `depends_on` 还是图结构推导？）
- 踩坑记录是独立节点还是节点的属性？
- 资源（文档、视频、源码）怎么组织？

**Step 2 — 用一个极简 HTML + localStorage 实现核心闭环**

实现最小闭环：
```
查看推荐节点 → 标记完成 → 数据持久化到 localStorage
```

这只需要一个 HTML 文件，不需要推荐算法，不需要图谱。目标是验证"记录 → 下次读取"这条路径是通的。

### 为什么先做这两步

1. **快速验证设计假设** — 实际填数据会发现 schema 遗漏或冗余
2. **给后续所有工作提供基准** — 前端、推荐引擎、图谱可视化都依赖同一个 schema
3. **风险最低** — 纯 YAML + 一个 HTML，改动代价极小

### 暂时搁置的方向

- 图谱可视化 UI — 等 schema 和基础闭环跑通再说
- 推荐算法 — 先用"按依赖顺序依次展示"的朴素策略
- 多项目支持 — Phase 1 专注单项目验证，Phase 2 再抽象
